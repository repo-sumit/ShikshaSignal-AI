# ShikshaSignal AI ‚Äî Monthly District Review Agent

> ‚öÝÔ∏è **SYNTHETIC DEMONSTRATION DATA ‚Äî NOT REAL GOVERNMENT RECORDS.**
> All data in this project is generated by a seeded script. District/block/school names are
> fictional. No child-level PII, no Aadhaar/APAAR, no real phone numbers, no scraped records.
> The schema is designed to *accept* real aggregate data (UDISE+/NAS-style) later via a documented mapping.

**ShikshaSignal AI** turns the scattered, mismatched reality of a district's education program data
(DIKSHA usage, FLN assessments, teacher training, field issues) into the **paste-ready review memo,
explainable risk ranking, and action tracker** a PMU analyst hand-builds before every DEO review ‚Äî
with **every number deterministic, traceable, and auditable**.

This repo is built in validated phases. The deterministic core (M1), the District
Review Compiler (M3), and free/local LLM providers (M4) are in. **Milestone 5 adds
an optional local Streamlit viewer for inspecting and downloading the generated
review artifacts** ‚Äî the CLI is unchanged and remains the source of truth; the
viewer never duplicates review logic.

---

## Design principle (carried into every phase)

> **The LLM never computes, ranks, or invents numbers.** KPIs, risk scores, rankings, and
> data-quality flags are produced by deterministic, unit-tested Python. The LLM (added in a later
> phase) only *narrates* numbers that this core already computed ‚Äî so the output is auditable and
> a wrong figure is impossible by construction.

## Run (Phase 1 ‚Äî deterministic core, no API keys, fully offline)

```bash
# 1) Generate the synthetic dataset (seeded ‚Üí reproducible)
python scripts/generate_synthetic_data.py --seed 42

# 2) Validate & inspect data quality (coverage %, planted bad rows, ID reconciliation)
python -m app.tools.data_quality

# 3) Print the district KPI summary (target-vs-actual-vs-last-period)
python -m app.tools.kpi_calculator

# 4) Compute decomposed, explainable risk scores
python -m app.tools.risk_score

# 5) Write the risk ranking + "what changed" to outputs/risk_ranking.csv
python -m app.tools.rankings

# Run the test suite (offline, hermetic)
pytest -q
```

The first command writes five CSVs to `data/synthetic/`. The risk-score command prints a believable
**~55% Low / ~30% Medium / ~15% High** band split and highlights one clear "problem block" ‚Äî the
earliest signal that the data and risk core are credible *before any AI is added*.

## Run (Milestone 3 ‚Äî Monthly District Review Compiler, offline-default)

```bash
# Compile the four review artifacts using the default offline MockLLM
python -m app.review --district "District Alpha" --period 2026-05
```

Writes:
- `outputs/monthly_district_review.md` ‚Äî the review memo
- `outputs/action_tracker.csv` ‚Äî proposed actions (status: `proposed`)
- `outputs/review_facts.json` ‚Äî the verified facts every memo number must trace to
- `outputs/audit_log.json` ‚Äî run lineage (provider, model, fallback flag, files)

## Run (Milestone 4 ‚Äî optional free / local LLM providers)

MockLLM remains the safest default ‚Äî no API keys, no network, deterministic, grounded
by construction. Real providers are wired behind the same interface and **never fail
the review**: missing credentials, network errors, or grounding violations all fall
back to MockLLM and are logged in `audit_log.json`.

```bash
# Offline default (no keys, no network) ‚Äî always works
python -m app.review --district "District Alpha" --period 2026-05 --llm-provider mock

# Google Gemini (free tier) ‚Äî needs GOOGLE_API_KEY (or GEMINI_API_KEY) in env
python -m app.review --district "District Alpha" --period 2026-05 --llm-provider gemini

# Groq (free tier) ‚Äî needs GROQ_API_KEY in env
python -m app.review --district "District Alpha" --period 2026-05 --llm-provider groq

# Local Ollama ‚Äî needs a running daemon at OLLAMA_BASE_URL (default localhost:11434)
python -m app.review --district "District Alpha" --period 2026-05 --llm-provider ollama

# Strict mode: any grounding failure rerenders the whole memo with MockLLM
python -m app.review --district "District Alpha" --period 2026-05 \
       --llm-provider gemini --strict-grounding
```

Environment variables (see `.env.example`):

| Variable | Used by | Default |
| --- | --- | --- |
| `LLM_PROVIDER` | global selector (overridden by `--llm-provider`) | `mock` |
| `GOOGLE_API_KEY` / `GEMINI_API_KEY` | Gemini | _(unset)_ |
| `GEMINI_MODEL` | Gemini | `gemini-1.5-flash` |
| `GROQ_API_KEY` | Groq | _(unset)_ |
| `GROQ_MODEL` | Groq | `llama-3.1-8b-instant` |
| `OLLAMA_BASE_URL` | Ollama | `http://localhost:11434` |
| `OLLAMA_MODEL` | Ollama | `qwen2.5:7b` |

### Safety guarantees (enforced by tests)

1. **The LLM never computes or invents numbers.** Every numeric token in the memo is
   checked against `review_facts.json` by `app.eval.grounding`; ungrounded numbers
   trigger a fallback for that section.
2. **The review never crashes because an optional provider is unavailable.** Missing
   keys or network errors fall back to MockLLM with `fallback_used=true` and a
   `fallback_reason` recorded in `audit_log.json`.
3. **Root causes stay labelled as hypotheses** requiring field verification.
4. **Actions always start as `proposed`.** Human approval is a downstream concern.
5. **No paid API is required for the project to run** ‚Äî MockLLM is feature-complete.

## Run (Milestone 5 ‚Äî local Streamlit viewer)

A small dashboard around the existing CLI. The viewer never duplicates review
logic ‚Äî it invokes `app.review.run_review(...)` (the same callable the CLI uses)
and reads the four artifacts back through `app.services.artifact_reader`.

```bash
pip install -r requirements.txt
python scripts/generate_synthetic_data.py --seed 42
python -m app.review --district "District Alpha" --period 2026-05 --llm-provider mock  # optional warm-up
streamlit run frontend/streamlit_app.py
```

The viewer is **local-first** and **synthetic-only**. It opens at
`http://localhost:8501`.

Sidebar controls:
- District (read from `data/synthetic/schools.csv`)
- Period (derived from synthetic weeks)
- LLM provider (`mock` / `gemini` / `groq` / `ollama` ‚Äî falls back to mock if unavailable)
- Top-N risky schools / blocks
- Strict-grounding toggle
- **Generate Review** button (runs the compiler end-to-end)

Main area:
- Executive overview ‚Äî `st.metric` cards driven by `review_facts.json`
- Five tabs: **Review Memo**, **Risk Ranking**, **Action Tracker**,
  **Audit Log**, **Review Facts**, each with a `st.download_button`

Guardrails preserved from earlier phases:
- All data remains synthetic and local-first.
- The LLM still never computes numbers; every memo number is grounded.
- Actions remain `proposed` until human approval.
- Missing data / missing credentials / provider fallback are all surfaced
  clearly in the UI ‚Äî never as a raw traceback (unless debug mode is on).

## Data dictionary (synthetic)

| File | Grain | Key columns |
|---|---|---|
| `schools.csv` | one row / school | `school_id`, hierarchy (`state‚Üídistrict‚Üíblock‚Üícluster`), `enrollment`, `teachers_count`, `internet_available`, `device_available`, `infrastructure_score` |
| `diksha_usage.csv` | school √ó week | `school_id`, `week` (ISO `YYYY-Www`), `qr_scans`, `sessions`, `learning_minutes`, `active_teachers`, `active_students_proxy` |
| `assessments.csv` | school √ó grade √ó subject √ó round | `assessment_round` (baseline/midline/endline), `baseline_score`, `current_score`, `district_average`, `proficiency_band` |
| `teacher_training.csv` | one row / teacher | `teacher_id`, `school_id`, `status`, `completion_percent`, `assessment_score`, `last_activity_date` |
| `field_issues.csv` | one row / issue | `issue_id`, `school_id`, `issue_type`, `severity`, `status`, `created_at`, `resolved_at` |

`school_id` is the single foreign key joining every fact table back to `schools.csv`.
