# Safety and Privacy

> ⚠ **Disclaimer:** ShikshaSignal AI is a demonstration project. It runs on **synthetic, seeded data only**. No real student, teacher, or school records are processed. The system does not connect to UDISE+, DIKSHA, NAS, Aadhaar, APAAR, or any government database. Nothing in this repo is an approved deployment, a government tool, or a production decision-making system.

This document explains the safety, privacy, and governance posture of the project — what it deliberately does, what it deliberately does **not** do, and what would have to change before real data could ever be touched.

---

## Synthetic-data policy

Every CSV under `data/` is generated by a seeded script:

```
python scripts/generate_synthetic_data.py --seed 42
```

- District, block, cluster, and school names are **fictional** (e.g. "District Alpha", "Madhopur block").
- No real records are scraped, sampled, or imported from any external system.
- The schema is intentionally shaped to **resemble** UDISE+/NAS/DIKSHA aggregate exports so that, **later**, a real aggregate feed could be mapped in via a documented adapter. That adapter does not exist today, and nothing in the current code path reads anything that isn't a freshly generated synthetic CSV.
- The data is described in memos and dashboards as **"synthetic DIKSHA-like data"** — never as real telemetry.

## No child-level PII

The CSVs aggregate up; they never carry a child.

- No student names.
- No dates of birth.
- No per-child marks or test scores.
- No per-child attendance rows.
- No per-child digital usage rows.

Where engagement matters, the system uses `active_students_proxy` — a **school-week aggregate**, not a per-student count. Learning outcomes are stored as `assessment_pct` at the **school × grade × subject × week** grain.

## No Aadhaar / APAAR / student IDs

Explicitly excluded by design. There is **no** Aadhaar number, APAAR ID, EMIS student ID, or any other persistent student identifier anywhere in the schema, the code, or the generator. None will be added.

## No real teacher records

- `teacher_id` values are synthetic strings of the form `TCH_NNNNNN`.
- No phone numbers.
- No addresses.
- No GPS coordinates.
- No biometric data.
- No employee IDs that map to any real human.

`teacher_training_status.csv` carries only the synthetic teacher ID, a synthetic school ID, the module name, and a completion percent.

## No automated outbound messages

Stakeholder messages are **drafted in the memo** (`outputs/monthly_district_review.md`) for a human reviewer to read, edit, and decide on.

The system has **no integration with**:

- WhatsApp / WhatsApp Business API
- SMS gateways
- Email (SMTP, SendGrid, etc.)
- Push notifications
- Telephony / IVR

Nothing leaves the local machine. The drafted stakeholder messages are text in a markdown file. A human has to copy, edit, and send them through their own approved channel.

## Actions are 'proposed', not approved

`outputs/action_tracker.csv` always starts every row at:

```
status = proposed
```

There is no code path that sets status to `approved`, `assigned`, `in_progress`, or `done` automatically. Approval is a **human action** taken outside the system. The compiler will never mark its own suggestions as accepted.

## Hypothesis labels

Every root-cause statement in the memo is labelled `**Hypothesis:**` and carries the phrase **"requires field verification."**

The model is not allowed to assert causation. It surfaces plausible explanations grounded in the numbers, and explicitly hands the burden of confirmation back to a human officer who can visit the block, talk to teachers, and verify on the ground.

## Human-approval principle

Anything that would touch a real person, a real school, or a real budget is gated by an **explicit human approval step**.

Today the code only does three things:

1. Reads synthetic CSVs.
2. Computes KPIs, risk scores, and rankings.
3. Writes four local artifacts: `monthly_district_review.md`, `action_tracker.csv`, `audit_log.json`, `review_facts.json`.

There are **no** integrations with payroll, HRMS, procurement, transfer orders, teacher rosters, school management systems, or any government workflow. Adding one would require a separate, reviewed design — not a config flag.

## Audit log

Every run writes `outputs/audit_log.json`. This is the artifact a reviewer or auditor reads to reconstruct exactly what happened. It records:

- `run_id`, `timestamp`, `command_args`
- `data_files_used` — every CSV read
- `policy_docs_used` — the policy map version consulted
- `llm_provider`, `requested_llm_provider`, `actual_llm_provider`, `model_name`
- `fallback_used` and `fallback_reason` — whether MockLLM took over, and why
- `grounding_failures` — any LLM output that referenced a number not in `review_facts.json`
- `provider_latency_ms` and per-section `section_metadata`
- `output_files` — every artifact written
- `risk_formula_version`, `risk_weights`
- `synthetic_data_notice`

If a memo cites a number, the audit log + `review_facts.json` together let a reviewer trace that number back to its source row.

## Future real-data considerations

If real aggregate data is ever integrated, the bar is:

- **Aggregate only** — district or block roll-ups. No school-level individuals, no per-child rows.
- **DPDP-style consent** — explicit, documented consent from the data owner (typically a state education department), with purpose limitation and retention limits written down.
- **Source documentation** — every field mapped to its source export, with the export's own data dictionary archived next to the adapter.
- **Reviewable adapter** — the mapping code reviewed by both an engineering and a policy reviewer before any real CSV is read.
- **No student-level data, ever.** Per-child records are out of scope by design and will remain so.

The current code is the demonstration; the policy above is the gate that any real-data work would have to clear first.

## Summary table

| Concern | What the system does | What it deliberately does NOT do |
|---|---|---|
| Data source | Reads seeded synthetic CSVs only | Reads no real UDISE+/DIKSHA/NAS/Aadhaar/APAAR data |
| Student records | Aggregates only (`active_students_proxy`, `assessment_pct`) | No names, DOBs, per-child marks, per-child attendance |
| Identifiers | Synthetic `SCH_*`, `TCH_*`, `BLK_*` IDs | No Aadhaar, APAAR, EMIS, phone, address, GPS |
| LLM role | Drafts narrative from `review_facts.json` only | Never computes numbers, never ranks, never invents facts |
| Numbers in memo | Every number traces to `review_facts.json` (grounding check) | Ungrounded numbers trigger MockLLM fallback or `--strict-grounding` re-render |
| Root causes | Labelled `Hypothesis`, with "requires field verification" | Never asserted as fact |
| Actions | Written to `action_tracker.csv` with `status=proposed` | No auto-approval, no auto-assignment |
| Outbound messaging | Drafts messages inside the memo for human review | No WhatsApp / SMS / email / push / IVR integration |
| External systems | None — fully local | No connection to HRMS, payroll, procurement, transfer orders |
| Auditability | Every run writes `audit_log.json` with provider, fallback, grounding, files | No silent runs; no undocumented model calls |
| LLM default | `mock` (Jinja templates, offline, no API key) | No paid API required to run the project |
